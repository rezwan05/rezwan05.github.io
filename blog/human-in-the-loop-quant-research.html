A Self-Improving Quant Research Harness with a Human in the Loop

A self-improving multi-agent workflow built for correctness: agents reason about what's holding a strategy back, test alternative fixes, and validate behavior with chart evidence across market regimes, while the human keeps direction and final judgment.

Feb 2, 2026 · 10 min read

In brief

  • AI coding agents made my research dramatically faster, including at producing convincing wrong answers. One early result reported more than eight times the real figure because of a subtle timing bug that crashed nothing.
  • My fix borrows from safety-critical engineering: two agents implement every hypothesis independently and must agree at the level of individual decisions, and I audit the behavior visually before believing any number.
  • The loop is self-improving. When a strategy underperforms, agents reason about why, propose alternative paths, and bring me chart evidence selected across market regimes, while the decisions that matter stay with me behind explicit evidence gates.

An early version of one strategy reported $2.23 million in profit over two years. After I corrected its fill timing, the result fell to $265,000.

Nothing crashed, and the equity curve looked great. The strategy had simply decided at each bar's close and filled at that same bar's open, which means it was trading on a price from five minutes before its decision existed.

That 88% haircut changed how I use coding agents for research. Implementation used to be the bottleneck, but an agent can now write a strategy faster than I can check it, so the real work has become deciding which results to believe.

The harness I ended up with spends most of its machinery on checking results rather than producing them. I define the hypothesis and the evidence that would settle it, Claude Code and Codex implement and analyze it, and deterministic code computes every number along the way. A candidate advances only when its code, its trades, and its behavior all pass.

The same machinery is what makes the loop self-improving. Each iteration diagnoses what isn't working, proposes alternative paths to fix it, and carries what it learned into the next attempt, so the harness gets smarter about a strategy the longer it works on it.

  1. Frame. I write a testable market hypothesis and its failure condition.
  2. Build. Two agents implement it independently in separate sandboxes.
  3. Review. Each agent reviews the other once; both address the findings.
  4. Measure. Deterministic backtests produce trade-level artifacts.
  5. Inspect. An agent selects representative charts, and I review the behavior.
  6. Diagnose. A regime-analysis agent reasons about where the idea works, where it fails, and why.
  7. Reproduce. Candidates that pass run in independent backtest engines and data paths.
  8. Improve. The agents propose paths to fix the weakest part, and I decide which to pursue.
The decision returns to the human before the next iteration.

I have written about pieces of this system before: the interactive chart tool, the data pipeline that turns 38 GB of ticks into backtests that run in minutes, and the 15-second timestamp bug that an independent data comparison exposed. This post is how the pieces fit together.

Start with a specific hypothesis

I don't start an agent with “find me a profitable strategy.” That prompt gives it too much freedom to change the question after seeing the answer.

Instead, I start with a mechanism. For example:

A trend-following strategy is vulnerable when its direction flips repeatedly in a short window. Block new entries after four flips in the trailing 30 signal bars, then compare it with the ungated strategy on the same periods.

Before implementation, I define what counts as a flip, when it becomes observable, when the next order can fill, the transaction costs, position limits, data windows, baseline, and pass criteria.

None of that is negotiable after the results come in. If a variant fails with one tick of slippage, the answer is that it failed, not that we should revisit the slippage model.

Implement it twice, independently

Claude Code and Codex collaborate through Codex MCP, but the first phase is deliberately non-collaborative. Each receives the same specification, data contract, and baseline. Each works in its own sandbox and cannot read the other's implementation until both have produced code, a result summary, and a trade log.

It's a small version of N-version programming. Agreement doesn't prove correctness, since both versions can share a wrong assumption, but disagreement is always informative. It means the spec was ambiguous, the data differs, or one of them has a bug.

In one experiment, both agents reproduced the baseline result to the cent: a loss of $38,761.24. Then their entries for the new strategy overlapped by only 23%, even though both implementations came from the same English spec on the same data.

Review found the difference in an afternoon. One version treated the trigger as a level and kept entering as long as the condition stayed true, which fired on 54.5% of eligible bars. The intended rule was an edge: enter only when the condition turns from false to true, which fired on 0.6%. A cross above a line is an event, not a state.

The identical baselines did real work here. They proved the engine, data window, and fill model were fine, so the hunt could skip all of that and go straight to strategy logic.

One review pass, then evidence

After both implementations are complete, each agent reviews the other once. The checklist is specific:

  • Is every feature based only on information available at that timestamp?
  • Are higher-timeframe values exposed only after the bar closes?
  • Are orders filled at prices that could have been traded?
  • Are commission, slippage, sizing, and session rules identical?
  • Does the code implement the written trigger, including its state transitions?
  • Can the headline result be recomputed from the raw trade log?

I cap this at one structured pass each. I've tried more, and the second pass mostly argues about style or reopens decisions the first pass already settled. So both agents do one pass, fix the findings, and rerun, after which I compare entry times, indicator values, fills, and independently recomputed P/L. If a difference stays unexplained, I mark the experiment inconclusive instead of arguing it to a verdict.

The process earns its keep on negative results too. In one review, an agent re-aggregated all 1,628 trades from the other implementation's raw trade log: identical timestamps, zero P/L difference. The idea genuinely didn't work. That's worth establishing carefully, because a false negative quietly buries a good idea.

Review the trades on charts

A summary can tell me a strategy made money. It can't tell me the strategy did what I meant.

After a run, an agent uses my backtest chart tool to produce a small audit set: strong winners, large losses, unusual holding periods, regime boundaries, and examples around the exact rule being tested. The selection is deliberate across market conditions, so I'm not just admiring trades from the periods that flattered the strategy. Each chart includes candles, entries, exits, stops, indicator values, volume, and hover details from the trade log.

I review those examples and ask plain questions. Did the stop move for the reason we intended? Did a higher-timeframe signal appear too early? Did a filter stay active through a trend, or switch on and off every few bars? Are repeated entries genuine opportunities or the same signal firing continuously?

Charts often expose rules that are mathematically valid but behave poorly. A chart may also suggest a new test. For example, I can compare an initial stop fitted to the current trend with the tightest available stop line. I test that as a new hypothesis instead of changing the current result by hand.

Analyze results by market condition

After the implementation passes review, a second agent groups the trades by session, volatility, trend strength, direction, holding time, and strategy state. It answers three questions:

  1. Where does the strategy earn its return?
  2. Where does it fail consistently?
  3. What is the smallest next experiment that could separate the two?

This table has changed my mind more often than any backtest summary. In one study, a trend strategy earned 84% of its P/L outside regular market hours even though its win rate was lower there. The overnight winners simply ran further. The same strategy, between 11:00 and 12:00 Pacific, had a profit factor of 0.42 and lost about $98,000 in the sample.

These results suggest new tests. They do not justify adding a time filter without walk-forward validation.

I use the same process for market regimes. In one study, ADX, Choppiness Index, and ATR percentile did not separate winners from losers. The number of recent strategy flips was more useful. That result led to a new controlled test of the churn gate.

Test the result in another engine

Two implementations can still share one engine's bug, so stronger candidates get rerun in independent systems:

  • a readable Python backtester is the research reference;
  • LEAN runs the same strategy against the same local market data;
  • QuantConnect cloud runs it on an independent data path;
  • TradingView and Pine Script provide an additional visual and execution-semantics check when appropriate.

I compare trade counts, timestamps, prices, costs, and P/L. Some differences are expected near contract rolls, session boundaries, and ambiguous intrabar fills. I investigate every unexplained difference.

For one combined strategy, the Python backtest produced $1.196 million. LEAN reproduced 96.5% of that result, and QuantConnect produced 102.4%, with trade counts within 4% across the three systems. That's the passing grade: the implementation transferred across engines and data paths. It did not prove profitability, because walk-forward evaluation was still pending. Correct implementation and a good strategy are separate questions.

Agreement has limits, though. One of the inflated early results had also been cross-checked, and an independent LEAN reimplementation matched it within 3% because both versions had faithfully implemented the same wrong fill assumption. Cross-engine agreement validates the implementation but not the assumptions behind it, which is why fill realism gets audited separately.

Use explicit promotion gates

Agents can test many ideas quickly. More tests also create more false positives, so each candidate must pass a fixed set of checks:

GateQuestionRequired artifact
BaselineDid the unchanged strategy reproduce on the same data and costs?Baseline summary and trade log
ImplementationDo the two implementations agree on the rule and its events?Code review, timestamp comparison, independent P/L aggregation
BehaviorDoes it act as intended on real market examples?Chart audit and regime report
GeneralizationDoes the improvement survive rolling out-of-sample windows after costs?Paired walk-forward results and trial count
PortabilityDoes it survive another engine or data source?LEAN, QuantConnect, or Pine reconciliation
PromotionDoes prospective behavior justify any exposure?Shadow results, limits, monitoring, and a human decision

These checks are the research-side view of a longer deployment ladder, which runs from static validity to tightly limited capital and gets its own post: Seven Gates Between an Idea and Live Money.

An experiment registry records the parent hypothesis, configuration, data snapshot, baseline, costs, result, and verdict for every run. Failed tests stay in the registry on purpose. They keep the total trial count visible, and they stop an agent from proposing the same idea again under a new name.

One strategy family went through about 250 configurations without producing a robust full-history result. That record is a useful output in itself: it tells me to spend research time elsewhere, and there is no configuration #251.

Keep key decisions with the human

At the end of an iteration, the agents return a short research summary:

  • what changed
  • what stayed identical to the baseline
  • where the result was strongest and weakest
  • which charts deserve review
  • what failed, and why
  • the smallest next hypothesis worth testing

The proposals are where the self-improvement lives. When a result disappoints, the agents don't just report the failure. They reason about why: which regime bled, which exits gave back profit, which signal fired too often. Each explanation becomes a candidate path framed as a testable hypothesis, and the registry stops them from re-proposing something that already failed under a new name.

I can approve the next test, change the question after reviewing a chart, or stop the work. My decision goes into the same registry as the agents' experiments, because a human cherry-picking results is still cherry-picking.

Inside a sandbox, an agent can write code, run tests, inspect trades, and prepare a proposal. It cannot change the cost model, redefine the baseline, access a sealed test window, promote its own result, or allocate capital.

This is a research system, not an autonomous trading bot.

What I learned

The main value comes from the checks around the agents:

  • Independent implementations make ambiguity visible.
  • One focused cross-review finds the real implementation issues.
  • Trade-level artifacts make every summary auditable.
  • Charts connect numbers to intended behavior.
  • Regime analysis turns surprises into testable follow-up questions.
  • Independent engines expose shared assumptions.
  • Promotion gates and a complete experiment log limit overfitting.

Agents changed the speed of the loop, but they didn't change the standard of evidence. If anything, the speed is exactly why the standard now has to be enforced by structure instead of willpower.

This describes an independent research workflow, not investment advice or a claim of live trading performance.
